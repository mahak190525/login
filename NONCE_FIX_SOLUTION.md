# Nonce Mismatch Error - Complete Solution

## The Problem

When using MSAL to authenticate with Microsoft and then calling Supabase's `signInWithIdToken()`, you get:

```
Error: Passed nonce and nonce in id_token should either both exist or not
```

### Why This Happens

1. MSAL generates an ID token with Microsoft's nonce
2. Supabase's `signInWithIdToken()` expects either:
   - Both the token AND the function call to have matching nonces, OR
   - Neither to have a nonce
3. The nonces don't match because:
   - MSAL's nonce is generated by Microsoft
   - Supabase expects its own nonce (from the OAuth flow it initiated)
4. Since we're using MSAL directly (not Supabase's OAuth), the nonces will never match

## The Solution: Supabase Edge Function

Instead of using `signInWithIdToken()`, we create a **Supabase Edge Function** that:

1. Receives the Microsoft ID token
2. Decodes it to get user info
3. Uses Supabase Admin API to create/update the user
4. Generates a Supabase session
5. Returns session tokens to the frontend

### Architecture

```

│   Browser   │
│   (MSAL)    │

       │ 1. Login with Microsoft
       │

       │

│   Browser   │

       │
       ▼

```

## What Gets Created in Supabase

When a user logs in, the Edge Function creates:

### In `auth.users` table:
- `id`: UUID (generated by Supabase)
- `email`: From Microsoft token
- `email_confirmed_at`: Auto-set (Microsoft verified it)
- `user_metadata`:
  - `full_name`: User's name from Microsoft
  - `avatar_url`: Profile picture from Microsoft
  - `provider`: "azure"
  - `provider_id`: Microsoft's user ID (sub/oid)

### Session tokens:
- `access_token`: For API calls
- `refresh_token`: For refreshing the session
- `expires_in`: 3600 seconds (1 hour)

## Troubleshooting

### Edge Function Not Deployed

**Error:** `Function not found` or 404

**Solution:**
```bash
supabase functions deploy exchange-azure-token
```

### Missing Service Role Key

**Error:** Edge Function can't create users

**Solution:** The Service Role Key is automatically available in Edge Functions as `SUPABASE_SERVICE_ROLE_KEY`. No action needed.

### Token Decode Error

**Error:** `Failed to decode Microsoft token`

**Solution:** Ensure you're passing the full ID token from MSAL response.

### No Email in Token

**Error:** `No email found in Microsoft token`

**Solution:** 
1. Go to Azure Portal → App registrations → Your app
2. Click "API permissions"
3. Ensure `email` scope is added and admin consented

## Comparison: Before vs After

### Before (with signInWithIdToken)

```javascript
// ❌ This causes nonce mismatch error
const { data, error } = await supabase.auth.signInWithIdToken({
  provider: 'azure',
  token: response.idToken,
  nonce: extractedNonce, // ❌ Doesn't match Supabase's expected nonce
})
```

### After (with Edge Function)

```javascript
// ✅ This works!
const exchangeResponse = await fetch(`${supabaseUrl}/functions/v1/exchange-azure-token`, {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${supabaseAnonKey}`,
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    idToken: response.idToken,
    accessToken: response.accessToken,
    provider: 'azure',
  }),
})

const sessionData = await exchangeResponse.json()

await supabase.auth.setSession({
  access_token: sessionData.access_token,
  refresh_token: sessionData.refresh_token,
})
```

## Security Considerations

### ✅ Secure

- Edge Function runs server-side (not exposed to client)

### ⚠️ Important Notes

1. **Email verification:** We auto-confirm emails because Microsoft already verified them
2. **Provider tracking:** We store `provider: 'azure'` in user_metadata
3. **Token validation:** Currently we decode without full validation (trusting MSAL did it)

## Summary

The nonce mismatch error is a fundamental incompatibility between MSAL-generated tokens and Supabase's `signInWithIdToken()` expectations. 

**The solution:** Use a Supabase Edge Function to handle token exchange server-side, creating users directly in the `auth.users` table and generating proper Supabase sessions.

**Result:** Microsoft login works perfectly, users are stored in Supabase, and you get all the benefits of Supabase Auth!
